# System-Wide Engineering Rules

This document outlines the strict operational and logical rules that the application must enforce during execution.

---

## 1. System Rules

### Stateless Execution
- The runtime application must not rely on local file persistence to preserve state between scheduled executions.
- Any state verification or reconciliation must be calculated dynamically by querying the current contents of the AI Vector Store and contrasting it with the source Help Center documentation.

### Idempotency Requirement
- Execution of the synchronization cycle under identical document states must result in zero modifications, zero API uploads, and zero deletions in the AI Vector Store.

### Delta Synchronization Enforcement
- The synchronization engine must compare current article fingerprints against historical sync states to isolate modifications.
- Only delta changes (newly created or modified articles) are permitted to trigger Vector Store upload operations. Unchanged documents must be skipped to conserve bandwidth and API rate allocations.

---

## 2. Data Rules

### Checksum Strategy
- The content signature of a normalized document must be generated by applying the SHA-256 algorithm to the finalized Markdown content string (excluding dynamic metadata timestamps).
- Checksums are used as the absolute comparator for detecting content drift.

### File Naming Rules
- Generated Markdown documents must be saved using deterministic and clean file names.
- Filenames must follow a consistent slug format derived from the article title (e.g., lowercase, alphanumeric characters, and hyphens replacing spaces).

### Markdown Formatting Rules
- Converted documents must start with a structured YAML front-matter metadata block.
- Markdown outputs must preserve heading hierarchies, lists, tables, and blockquotes, while stripping HTML style tags and scripts.
- Image tags must be formatted as standard Markdown images.
- Relative hyperlinks referencing other help center articles must be resolved and written as absolute URLs pointing back to the live support portal.

---

## 3. Synchronization Rules

### Sync Log Classification
Every article discovered in the ingestion queue must be mapped to exactly one of the following sync states:
- **`added`**: The article is not present in the historical sync state or Vector Store; triggers a new file upload.
- **`updated`**: The article ID exists but the content hash differs from the recorded state; triggers a content replacement upload.
- **`skipped`**: Both the article ID and content hash match the sync state; no action is performed.
- **`deleted`**: An article ID exists in the sync state but is absent from the live Help Center queue; triggers file removal from the Vector Store.

### Failure Handling
- If an individual article encounters a processing error (e.g., conversion failure or single upload timeout), the sync engine must log the failure, flag the article as failed, and proceed with processing the remaining documents.
- In the event of a partial failure, the synchronization run must log the counts of succeeded and failed files, ensuring an error summary is written to stdout.

---

## 4. AI Assistant Rules

### Source Grounding
- The AI Assistant is strictly prohibited from generating responses based on general pre-trained weights if the information is absent from the uploaded documentation.
- The Assistant must answer queries using *only* the retrieved knowledge context.

### Hallucination Prevention
- If the retrieved context is insufficient or does not contain the answer, the Assistant must explicitly state that the information is unavailable in the support documentation and decline to guess.
- The Assistant must not combine retrieved documentation with unsupported assumptions.

### Citation Enforcement
- Every query response must include explicit citations pointing to the source articles.
- The Assistant must cite up to 3 official article URLs at the end of its response.

---

## 5. Operational Rules

### Docker Single-Run Behavior
- The containerized application must run the sync cycle once and terminate immediately upon completion.
- Successful executions must exit with status code `0`. Execution failures (e.g., API authentication failures or invalid configurations) must terminate with a non-zero exit code to signal alerting systems.

### Logging Requirements
- All logs must be written to standard output (`stdout`).
- Execution metrics (status, added, updated, skipped, deleted) must be printed at the end of the execution run in a structured, easily parseable block.
- No sensitive credentials, API keys, or security tokens are permitted to be written to logs.
